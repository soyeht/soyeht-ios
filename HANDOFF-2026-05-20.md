## Architecture Proof Delivered

**Cross-platform Bonjour interop on Linux was broken (`mdns-sd 0.10` emits 0 UDP 5353 packets, confirmed via `tcpdump` on devs).**

Fixed via two complementary changes:

1. **mdns-sd bump 0.10 → 0.13.11** (Cargo.toml workspace dep + Cargo.lock). Restores some broadcast capability but still incomplete (announce records appear empty in tcpdump capture).

2. **Pair-device URL `host=` fallback** (the architectural Apple-grade move):
   - Engine `install_cli` includes `&host=<tailnet-ipv4>:<port>` in pair-device URI when its own Tailnet endpoint is known.
   - iPhone `PairDeviceQR` parses the optional `host` field; `HouseholdPairingService.directEndpoint` short-circuits Bonjour browse and posts directly to the URL-embedded address.
   - Bonjour discovery remains the gold path when both peers' mDNS stacks agree (Mac↔iPhone). The `host` field is a safe fallback that inherits the same trust model (hh_pub in URL is verified by PairingProof regardless of how endpoint was reached).

## Remaining 9 Flows — Implementation Plan

### Apple-Grade Auto-Discovery Path (Chosen by Caio)

The 70% of infra is already in the engine:

- `setup_beacon.rs` publishes `founder_candidate` / `member_candidate` (TXT roles per spec 003)
- `bonjour_browser.rs spawn_bonjour_browser` is the founder-side scanner that picks up candidates
- `handlers_owner_events.rs` long-polls owner approvals
- `handlers_pair_machine.rs` mints `MachineCert` on owner approval
- Candidate-side `pre_household_router` exposes `local/seed` + `local/finalize` for cert reception

What's missing:

1. **Engine auto-become-candidate at startup** (Rust, ~3h)
   - Refactor `install_cli::run_pair_machine` into a library fn `auto_become_pair_machine_candidate(state_dir, transport, hostname, policy)` that returns a JoinHandle
   - In `household_bootstrap.rs` startup: after `determine_role` returns `MemberCandidate` (existing household visible on Tailnet) AND no local identity AND no active `pair_machine_window` → invoke the helper as a background task
   - Engine itself becomes the candidate without requiring `theyos install --pair-machine` CLI invocation

2. **Mac.app "Looking for home" Welcome variant** (Swift, ~2h)
   - Detect via `HouseholdBonjourBrowser` on app launch that another household is reachable on Tailnet
   - Show a polling screen with the discovered household name + "approve on the iPhone/Mac that already lives here"
   - When `/bootstrap/status` reports identity loaded, transition to main UI
   - When user explicitly wants to be a founder instead, "Create a new home" button bypasses the candidate flow

3. **iPhone owner approval card polish** (Swift, ~1.5h)
   - `OwnerEventsLongPoll` already surfaces `JoinRequest`s. Verify the existing UI card shows 6-word BIP-39 fingerprint and triggers Face ID on Confirm.
   - Wire the confirmation back through `handlers_pair_machine::confirm` so engine signs the MachineCert and pushes to candidate's `local/finalize`.

4. **No-limbo polish on iPhone Home** (Swift, ~1h)
   - After successful pair to a Linux-founder household, iPhone lands on `HouseholdHomeView` which only shows identity strings + gear + scanner. Needs an empty-state CTA ("Start your first session" or similar) when the household has zero conversations. This is independent of Linux specifically; same issue exists for any household with no live terminal sessions.

### Sequence to Validate 12 Flows

Per-flow reset cost is ~5 min (state wipes on all 3 devices). Suggested validation order maximizes reuse of household state:

1. **Linux→iPhone→Mac chain** (Flow 11): extends today's Flow 10 result — already have Linux+iPhone, only need Mac to join. Validates Mac.app auto-candidate path.
2. **Linux→Mac fresh** (Flow 9): reset all, Linux founder, Mac auto-joins as solo owner.
3. **Linux→Mac→iPhone chain** (Flow 12): same setup as Flow 9, then iPhone joins.
4. **Mac→Linux fresh** (Flow 6): reset all, Mac founder, Linux runs `theyos install --pair-machine` (CLI) → iPhone owner... wait, no iPhone yet. This means Mac alone approves Linux via `_owner/events` long-poll. Mac.app needs the owner-approval UI too. Add to scope.
5. **Mac→Linux→iPhone chain** (Flow 8).
6. **Mac→iPhone→Linux chain** (Flow 7).
7. **iPhone→Linux fresh** (Flow 2): reset all, iPhone has no engine to run pair-machine. This flow needs iPhone to **drive** Linux installation. User runs install on Linux, Linux engine ends up uninitialized, iPhone discovers via Bonjour, Caso-B-on-Linux flow kicks in. Engine `_soyeht-setup._tcp.` setup_role=founder_candidate gets picked up by iPhone `SetupInvitationBrowser` (currently Mac-only; need to extend for iPhone).
8. **iPhone→Linux→Mac chain** (Flow 4).
9. **iPhone→Mac→Linux chain** (Flow 3): iPhone→Mac is Flow 1 done; adding Linux via auto-candidate.
