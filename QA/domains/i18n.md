# Internationalization (15 languages)

## Objective
Verify that every user-facing surface renders correctly in all 15 supported
languages — en (base), pt-BR, pt-PT, es, de, fr, ru, id, ar, ur, ja, hi, mr,
te, bn — without layout breakage, missing keys, or untranslated strings.

## Risk
- A string referenced in code was never added to `.xcstrings` → user sees the
  raw key or English fallback.
- RTL languages (ar, ur) mirror the layout in ways custom AppKit views don't
  anticipate.
- French/German expansion (typically +30% vs English) overflows fixed-width
  buttons in the Welcome cards and Claw Store cards.
- The three catalogs (iOS app, macOS app, SoyehtCore) drift independently —
  coverage tests catch missing translations in any of the three.

## Preconditions
- Latest build of both Soyeht (iOS) and SoyehtMac.
- No real server required — smoke tests cover catalog resolution only.

## Fixtures
- Xcode Scheme → Run → Options → App Language — used to force a runtime locale
  without changing system preferences.
- `TerminalApp/Soyeht/Localizable.xcstrings` — iOS catalog.
- `TerminalApp/SoyehtMac/Localizable.xcstrings` — macOS catalog.
- `Packages/SoyehtCore/Sources/SoyehtCore/Resources/Localizable.xcstrings` —
  shared catalog (APIError, ClawNotificationHelper, UnavailReason, ColorTheme).

## Known gaps / follow-ups

- **Native-speaker review**: strings for mr, te, bn were machine-translated by
  the migration author and flagged `state: "needs_review"` in the catalog. The
  coverage test accepts both `translated` and `needs_review` for that reason.
  Before a public shipment targeting those locales, route them through native
  reviewers and flip the state to `translated`.
- **RTL layout audit**: ar and ur run through SwiftUI's automatic mirroring,
  but custom AppKit containers (e.g. the QR handoff popover, Welcome cards) use
  explicit anchor constants that don't flip. If the product decides to ship
  to ar/ur markets, schedule a pass with `NSApp.userInterfaceLayoutDirection
  = .rightToLeft` in a debug override.
- **Terminal output**: strings that travel from the server (`APIError.http`'s
  server-supplied body, `UnavailReason.installFailed`'s `%@` error) are not
  localized by the iOS / macOS catalogs — they come through as whatever the
  server writes. A future PR should either localize server errors or mark them
  `// i18n-exempt` so the distinction is deliberate.
